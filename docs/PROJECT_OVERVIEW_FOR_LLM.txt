1. WHAT CHECKMATE IS
================================================================================

One-sentence pitch:
  CheckMate is a social app that gives every person — regardless of what they're
  working toward — a daily trigger to document and share goal progress, combining
  BeReal's accountability mechanic with Strava's community momentum, for any goal.

Mechanic:
  Users set personal goals → daily randomized notification (5am–10pm local) →
  commit to a "promise time" to post a photo → share progress with followers and
  goal communities. Think: BeReal + Strava social layer, for ANY life goal.

Problem:
  Existing apps are goal-type-specific (Strava=fitness, MFP=diet, LinkedIn=career,
  BeReal=no goal context). CheckMate is category-agnostic accountability + social.

================================================================================

  Mobile:     Expo SDK (React Native), Expo Router, NativeWind
  Web:        Next.js 14 App Router, Tailwind, Server Components by default

  Errors:     Sentry

Monorepo:
  apps/web/              Next.js UI + API routes (apps/web/app/api/)
  apps/mobile/           Expo app
  packages/shared/       Zod schemas, types, constants
  packages/ui/           Shared UI components
  packages/server/       Business logic (DB, feed, notifications)

  supabase/migrations/   SQL migrations (create new files; don't edit old ones)

3. CRITICAL BUSINESS RULES (NEVER BREAK)
================================================================================

1. Streak logic is ALWAYS server-side (Inngest at midnight local per user).
   Never calculate streaks in API routes or UI components.

2. is_late is set server-side when leeway_expires_at passes and posted_at is
   null or after leeway.

3. streak_credited is true only if posted_at is before leeway_expires_at.

4. Trigger time is randomized 5am–10pm in user's timezone, calculated server-side
   at midnight each day.

5. Promise time must be set within 2 hours of trigger_fired_at (server validation).

6. leeway_expires_at = promise_time + 1 hour (always server-calculated).

7. Max 5 active goals per user — enforce at API level, not UI only.

8. Leaderboard data is read from leaderboards table (denormalized). Never compute
   on the fly in user-facing requests.

9. All photo URLs are signed Supabase Storage URLs. Never public bucket URLs.

10. Regional leaderboards fall back to global if < 3 users in region + category.

11. Do not show leaderboards with fewer than 3 community members.

Top 3 rules (short form):
  - Streaks = Inngest only
  - Max 5 active goals at API
  - Photos = signed URLs only

  id, post_id, user_id, body, created_at, deleted_at

  follower_id, following_id, created_at

setting deleted_at (see supabase/runbooks/apply_posts_soft_delete_rls.sql).

5. API & CODE CONVENTIONS
================================================================================

API routes: apps/web/app/api/ (Next.js Route Handlers, not pages/api)

Request validation: Zod schemas in packages/shared/src/schemas/

Error shape: { error: string, code: string }

List endpoints: cursor + limit pagination

JSON: camelCase in responses

Auth: Supabase session via createServerClient (@supabase/ssr) on protected routes

Supabase patterns:
  Server: createServerClient from @supabase/ssr
  Client: createBrowserClient from @supabase/ssr
  Never use service role key in client code
  Prefer RLS over manual auth checks in routes
  Service role (createAdminClient): server-only mutations when needed

Code style:
  TypeScript, no any
  Shared types: packages/shared (do not redefine)
  async/await, explicit errors, no silent catches
  One component per file; filename = component name
  No console.log in committed code

What NOT to do:
  - getServerSideProps / getStaticProps (App Router only)
  - Redux (use Zustand)
  - styled-components / emotion (Tailwind / NativeWind)
  - Raw SQL in app code
  - Streak/leaderboard math in API routes
  - fetch() for Supabase (use client)
  - Hardcoded timezone on client (use user.timezone from DB)
  - > 5 active goals

When unsure: ask before inventing; prefer simple v1 solutions; TODO over guessing.

8. USER JOURNEYS & PRODUCT SCOPE (V1)
================================================================================

Ship exactly v1 scope below. Nothing more for launch.

Journey 1 — Onboarding & first goal
  Success: account, ≥1 goal with category, promise time, personal feed
  Failures: social login → email/phone; skip goal → re-prompt; skip promise →
  default 8pm local; duplicate account → merge or clear error

Journey 2 — Daily trigger → promise → post
  Success: trigger in 5am–10pm window, promise set, photo in window+leeway,
  streak increments, post in feeds
  Failures: no push → in-app banner; miss leeway → late post, streak reset;
  no promise → 10pm deadline; upload fail → retry; no goals → prompt create

Journey 3 — Social feed & reactions
  Success: feed from follows + communities, react, comment
  Failures: empty feed → community default; offline → cached read-only;
  toxic content → report (manual review v1)

Journey 4 — Goal community discovery
  Success: browse/search categories, join, see posts
  Failures: no results → suggest category; empty → invite prompt

Journey 5 — Streak & leaderboard
  Success: streak per goal, weekly/all-time global+regional ranks
  Failures: broken streak → explain miss; <3 community → hide board;
  <3 regional → global only

Journey 6 — Follow user
  Success: follow from profile, posts in feed; block/report available
  v1: all profiles public by default

Journey 7 — Profile
  Success: goals, post grid, streaks, leaderboard rank
  Failures: no posts → empty grid; archived goal hidden publicly, posts kept

Roadmap beyond v1:
  V2: adaptive triggers, privacy, contacts, contextual ads, intl auth
  V3: sponsor leaderboard prizes, anti-cheat, sponsor admin
  V4+: AI coaching, certificates, brand marketplace

9. DAILY TRIGGER & PROMISE SYSTEM
================================================================================

V1 — randomized local window:
  - Random time 5:00 AM – 10:00 PM user local timezone, per user per day
  - Randomization server-side at midnight local, enqueued via Inngest

V2 (deferred): adaptive timing from post history

Promise time:
  - Set within 2 hours of trigger_fired_at
  - If unset, 10:00 PM local is hard deadline
  - leeway_expires_at = promise_time + 1 hour
  - Post before leeway: on-time, streak_credited
  - After leeway: still can post, marked late, no streak credit
  - Per goal, per day

Why vs BeReal: user commits to when they'll post (agency), leeway handles real life.

Technical:
  - Store timezone at signup (device-detected, adjustable)
  - daily_challenges holds trigger_fired_at, promise_time, leeway_expires_at,
    posted_at, is_late, streak_credited
  - Streak job at midnight local only

11. AUTH, SCREENS, NON-FUNCTIONAL REQUIREMENTS
================================================================================

Auth V1: email magic link, Google OAuth, Apple Sign In, phone SMS OTP
Deferred V2: KakaoTalk, WeChat, WhatsApp

Permissions V1:
  Push — after first goal
  Camera / photo library — on first post tap
  No location (use IP/device for timezone), no contacts v1

Screen flow (tab bar):
  Home Feed → Discover → [+] Post → Streaks/Leaderboard → Profile
  Post flow: trigger status, promise time, camera/library, tag goal, caption

Performance targets:
  Feed < 1.5s on 4G; upload < 5s for 5MB; trigger push within 60s of schedule;
  leaderboard hourly refresh; promise countdown real-time in app

Security / privacy:
  HTTPS; signed photo URLs; GDPR/CCPA deletion endpoint; age gate 13+ / 16+ EU;
  no ad targeting under 18; PII via Supabase encryption

Analytics V1 (PostHog):
  DAU, retention, trigger→promise, promise→post, late rate, streak distribution,
  category split. No precise location/contacts.

Ads: none in V1. V2 contextual by goal category. V3 sponsor prizes on leaderboards.

Support V1: in-app report → email; Sentry crashes; no live chat

Offline V1: no offline posting; cache feed for read; offline banner

Platform: web ships first; mobile TestFlight/internal track target

14. DEPLOYMENT & PRODUCTION OPS
================================================================================

Deployable pieces:
  Database/Auth/Storage — Supabase (migrations + runbooks)
  Web + API + /api/inngest — Vercel (root directory: apps/web)
  Background jobs — Inngest Cloud (sync serve URL to web)
  Mobile — Expo EAS (later)

Supabase production:
  1. Run all supabase/migrations/ in SQL Editor
  2. Run runbooks as needed (messaging, notifications, soft-delete RLS)
  3. Auth → URL Configuration:
     Site URL: https://your-app.vercel.app
     Redirect: https://your-app.vercel.app/auth/callback and /**
  4. Google OAuth redirect stays: https://nfpeasuabkwobyvocecc.supabase.co/auth/v1/callback
  5. SMTP recommended (built-in email rate limit ~2/hr)
  6. Storage: post-images bucket + policies
  7. Email alerts: RESEND_API_KEY, EMAIL_FROM on Vercel

Vercel setup:
  Import dpark1719/CheckMate, root apps/web, include files outside root (monorepo)
  Env (production):
    NEXT_PUBLIC_SUPABASE_URL
    NEXT_PUBLIC_SUPABASE_ANON_KEY
    SUPABASE_SERVICE_ROLE_KEY  (server only — required for some admin paths)
    NEXT_PUBLIC_APP_URL        (no trailing slash)
    INNGEST_EVENT_KEY, INNGEST_SIGNING_KEY
    NEXT_PUBLIC_POSTHOG_KEY/HOST (optional)
    UPSTASH_REDIS_* (optional)
    RESEND_API_KEY, EMAIL_FROM (email notifications)
  After deploy: set NEXT_PUBLIC_APP_URL to real URL, redeploy, update Supabase redirects

Inngest:
  Serve URL: https://your-app.vercel.app/api/inngest
  Functions discovered from inngest/

Mobile production env (apps/mobile/.env):
  EXPO_PUBLIC_SUPABASE_URL
  EXPO_PUBLIC_SUPABASE_ANON_KEY
  EXPO_PUBLIC_API_URL=https://your-app.vercel.app
  Supabase redirect: checkmate://auth/callback (match app.json scheme)

Launch checklist:
  [ ] All migrations on prod Supabase
  [ ] NEXT_PUBLIC_APP_URL matches live URL
  [ ] Supabase redirect URLs include /auth/callback
  [ ] Google OAuth tested on prod URL
  [ ] SMTP if using email auth heavily
  [ ] Inngest synced
  [ ] SERVICE_ROLE_KEY only server-side, never NEXT_PUBLIC_*

Troubleshooting:
  Vercel build fail → include files outside root; apps/web/vercel.json
  Auth redirect error → Supabase URLs must match NEXT_PUBLIC_APP_URL/auth/callback
  Google works local not prod → add prod URL to Supabase + consent screen users
  Google Internal Server Error → redeploy auth/callback; fix APP_URL + redirects
  Jobs never run → Inngest URL + INNGEST_* keys
  Delete post fails → run apply_posts_soft_delete_rls.sql; ensure SERVICE_ROLE on Vercel
  Messaging fails → run apply_messaging.sql; check /api/conversations/health
